# NocoBase + бот VL: внедрение

Интеграция в коде: [src/nocobase.js](../src/nocobase.js), вызовы из [src/index.js](../src/index.js) (webhook оплаты, выдача прокси, каталог в `/api/me`).

## 1. Безопасность инстанса (обязательно до продакшена)

Текущий доступ по `http://IP:10200/` без TLS — **не годится** для админки и API-ключей.

- **Домен + HTTPS**: поставьте reverse-proxy (Caddy или Nginx + Let’s Encrypt) на 443, бэкенд NocoBase оставьте на localhost.
- **Firewall**: снаружи открыть только `80/443`; порт NocoBase не публиковать в интернет, если прокси на той же машине.
- **Доступ админов**: по возможности allowlist IP / VPN; сильные пароли, отдельный пользователь для API.
- **Плагин API Keys** в NocoBase: отдельная роль с минимальными правами только на нужные коллекции ([документация NocoBase по API keys](https://docs.nocobase.com/integration/api-keys/usage)).
- **Бэкапы Postgres**: ежедневный дамп БД NocoBase (pg_dump) + хранение вне сервера.

После этого в `.env` бота укажите **`NOCOBASE_BASE_URL=https://ваш-домен`** (без завершающего `/`).

### subscription_branding (подпись подписки в VPN-клиенте)

Поля из NocoBase **сами по себе не попадают** в Happ / v2rayN / Hiddify: клиент читает **ответ подписки с 3X-UI**. Чтобы название узла в списке серверов совпадало с «Заголовком подписки» из вашей таблицы, бот при выдаче XUI-клиента пишет в панель поле **`remark`** (и при необходимости обновляет его).

Создайте коллекцию (имя по умолчанию **`subscription_branding`**, переименование: `NOCOBASE_COLLECTION_SUBSCRIPTION_BRANDING`):

| Поле | Тип | Примечание |
|------|-----|------------|
| subscriptionTitle | Строка | Пойдёт в `remark` клиента 3X-UI (видно в клиенте после обновления подписки) |
| supportUrl | Строка | Показывается в мини-аппе (ссылка «Поддержка») |
| profileUrl | Строка | Показывается в мини-аппе |
| announcement | Многострочный текст | Показывается в мини-аппе |
| active | Булево | Не `false` — строка участвует в выборе |

##### Пустая строка в блоке и только колонка «Действия»

Так бывает в двух случаях (часто **оба сразу**):

1. **В коллекции не созданы поля** (или у полей другое **внутреннее имя**, не `subscriptionTitle` / `supportUrl` / …). Импорт **не добавляет колонки сам** — он только подставляет значения в уже существующие поля. Если полей нет, запись может появиться **пустой**.
   - Зайдите в **менеджер источников данных** → **Коллекции** → **`subscription_branding`** → **Поля** и создайте все поля из таблицы выше.
   - При создании каждого поля задайте **имя поля (ключ в API / в импорте)** **точно** как в таблице: `subscriptionTitle`, `supportUrl`, `profileUrl`, `announcement`, `active`. Подпись в интерфейсе может быть по-русски — это не мешает.
2. **На странице в блоке таблицы не включены колонки**. В редакторе страницы откройте **настройки блока** (таблица `subscription_branding`) и в списке колонок **добавьте отображение** `subscriptionTitle`, `supportUrl`, … — иначе видна только служебная колонка **Действия**.

**Что сделать сейчас:** удалите пустую запись → создайте поля в коллекции → **скопируйте первую строку** из **Экспорт → Excel** (или из шаблона импорта) этой коллекции в **первую строку** листа `subscription_branding` в `Товары.xlsx` → снова `python import/xlsx_to_products_csv.py` → импортируйте **`nocobase-subscription_branding.xlsx`**. Заголовки в файле и в NocoBase будут совпадать один в один. Запасной пример строки заголовка + данных: [`import/subscription_branding.example.csv`](../import/subscription_branding.example.csv) (если ваши ключи полей такие же).

Берётся **активная** строка с заполненным `subscriptionTitle`, иначе любая активная, иначе последняя по `id`. Кэш NocoBase ~45 с.

Шаблон `remark` (опционально): переменная окружения **`XUI_CLIENT_REMARK_TEMPLATE`**, плейсхолдеры `{subscriptionTitle}` `{telegramId}` `{username}`. Если не задано: `{subscriptionTitle} · {telegramId}` или `VL · {telegramId}`.

После смены текстов пользователь в мини-аппе нажимает **«Обновить ссылку (XUI)»**, затем в VPN-клиенте — **обновить подписку**.

**Заголовок группы подписки** (как имя профиля рядом с доменом) часто задаётся в **самой 3X-UI** (настройки подписки / subscription title), а не через эту таблицу — при необходимости продублируйте там же.

#### Импорт из Excel (`Товары.xlsx`)

Один файл **`Товары.xlsx`** в **корне репозитория** — в нём два смысловых блока:

| Лист в Excel | Назначение | Выход скрипта |
|----------------|-------------|----------------|
| **`Data`** (если есть; иначе берётся активный лист, но не `subscription_branding`) | Коллекция **Товары** / `products` | `import/products-nocobase.csv` и `.xlsx` |
| **`subscription_branding`** (имя листа **строго** так) | Коллекция **subscription_branding** | только **`import/nocobase-subscription_branding.csv`** и **`.xlsx`** (копия листа без переименования колонок) |

Перед сохранением `Товары.xlsx` переключитесь на лист **`Data`**, чтобы активным не остался только брендинг — иначе скрипт остановится с подсказкой.

**Строка 1 листа `subscription_branding`** должна быть **точно как в выгрузке NocoBase** для этой коллекции (те же подписи/ключи, что в «Экспорт Excel» или в шаблоне импорта). Скрипт **не подменяет** заголовки — переносит лист в `nocobase-subscription_branding.*` как у товаров. Пустые строки данных пропускаются; пустые имена колонок в строке 1 игнорируются.

1. Установите зависимость и пересоберите CSV/XLSX для импорта.

   **Из корня репозитория** (`бот ВПН`, там где `package.json`):

   ```text
   python -m pip install -r import/requirements.txt
   python import/xlsx_to_products_csv.py
   ```

   Если вы уже в папке **`import`** (в приглашении cmd видно `...\бот ВПН\import>`), команды такие — **без** префикса `import\`:

   ```text
   python -m pip install -r requirements.txt
   python xlsx_to_products_csv.py
   ```

   Или из **`import`** подняться на уровень вверх и вызвать как в первом блоке:

   ```text
   cd ..
   python -m pip install -r import/requirements.txt
   python import/xlsx_to_products_csv.py
   ```

   **Windows:** можно дважды щёлкнуть **`import/regenerate-products-csv.bat`** — он переходит в корень проекта и вызывает `pip` + скрипт (в файле только латиница: кириллица внутри `.bat` часто ломает `cmd.exe`). Если не сработало — используйте команды из блоков выше вручную.

2. После запуска создаются файлы:
   - **`import/products-nocobase.csv`** (UTF-8 с BOM) и **`import/products-nocobase.xlsx`**;
   - при наличии листа **`subscription_branding`**: **`import/nocobase-subscription_branding.csv`** и **`.xlsx`** (лист **Sheet1** в xlsx).

3. В NocoBase: **Импортировать** → для **Товаров** — `products-nocobase.*`, для **`subscription_branding`** — **`nocobase-subscription_branding.xlsx`** (в этой коллекции, не в Товарах). Чтобы не плодить дубли тарифов, используйте **обновление по `code`**.

##### Ошибка: «Заголовки не найдены… ID, code, Целое число, productType…»

Это шаблон коллекции **Товары** (`products`). Файл **`nocobase-subscription_branding.xlsx`** нужно загружать в мастере импорта **у коллекции `subscription_branding`**: в админке откройте именно эту коллекцию → действие **Импорт** → выберите файл. Если сначала открыть «Товары» и там нажать «Импорт», NocoBase будет ждать колонки тарифов, а не брендинга.

Надёжный вариант: в коллекции **`subscription_branding`** сделайте **Экспорт → Excel** (даже пустой таблицы), скопируйте **первую строку** в лист `subscription_branding` файла `Товары.xlsx`, подставьте данные ниже, пересоберите `nocobase-subscription_branding.xlsx` скриптом — заголовки совпадут с ожиданиями импорта.

Заголовки **товаров** как у экспорта NocoBase: **`ID`, `code`, `Целое число`, `productType`, `title`, `sortOrder`, `active`, `serverId`**. Поле **`grantDays`** в таблице называется **«Целое число»**. Колонка **`serverId`** может быть пустой — сначала **создайте поле `serverId` (строка)** в коллекции через **«Поля»**, иначе при импорте уберите столбец из файла или импортер выдаст ошибку.

##### Почему не появились «новые колонки» и задвоились строки

- **Новые колонки** в NocoBase **не создаются** из файла импорта. Поля задаются только в админке: коллекция **Товары** → **Поля** → добавить поле (например **`serverId`**, тип **строка**). Импорт лишь **заполняет** уже существующие колонки.
- **Шесть одинаковых строк** — типичный случай: в коллекции уже были 3 тарифа, импорт **добавил ещё 3** как новые записи, а не обновил старые. Что сделать:
  1. Удалить **лишние три** дубликата вручную или отфильтровать по дате создания;
  2. В следующий раз в мастере импорта включить режим **обновления по уникальному полю** **`code`** (или как называется в вашей версии: *update / upsert / merge*), **либо** перед импортом очистить таблицу, **либо** импортировать в пустую коллекцию.

| Поле | Тип | Примечание |

## 4. Поведение бота (фаза A)

### Мини-приложение: ссылка на оплату

В `.env` бота (см. [.env.example](../.env.example)):

- **`PAYMENT_CHECKOUT_URL_TEMPLATE`** — если задан, кнопки «Продление» открывают этот URL в браузере (Telegram `openLink`). Подстановки: `{telegramId}`, `{productCode}`, `{grantDays}`, `{username}` (email/логин клиента в XUI, может быть пусто).
- **`PAYMENT_DEFAULT_PRODUCT_CODE`** — для экрана без привязанного аккаунта (кнопка «Оплатить / Продлить»), по умолчанию `vpn_30`.

Если шаблон **пустой**, а **`TEST_GRANT_ENABLED=1`** — сохраняется тестовое продление через API. Если оба выключены — пользователю показывается подсказка обратиться в поддержку.

После успешного **`POST /api/webhooks/payment`** (секрет `PAYMENT_WEBHOOK_SECRET`):

1. Выполняется `xuiProvisionCore` (как раньше).
2. Асинхронно создаётся/обновляется **customer** и запись **order** (поля из тела webhook).

Дополнительные поля тела webhook (опционально): `amount`, `currency`, `externalPaymentId` или `paymentId`, `productCode`, `username`, **`feeAmount`**, **`netAmount`**, **`serverId`** (см. таблицу `orders` выше).

После **`POST /api/proxy/provision`** в NocoBase пишется строка **proxy_instances** (без пароля).

Ошибки NocoBase **не роняют** выдачу VPN/прокси — только логируются в консоль.

## 5. Дашборды в NocoBase (аналитика, графики)

Страница **«Аналитика»** (или аналог): блок **Графики** + таблицы.

### Частая ошибка: график «не про деньги»

Если на оси категорий оказались значения **30 / 90 / 180**, а в подсказке метрика **`grantDays`**, вы построили диаграмму **по длительности тарифов**, а не по оплатам. Столбцы будут совпадать с подписями «на глаз» — это **не выручка**.

**Нужно для денег:** коллекция **`orders`**, мера **`sum(amount)`** или **`sum(netAmount)`**, измерение по времени — **`paidAt`** (день/месяц), фильтр **`status`** = paid (если используете несколько статусов).

### Выручка (orders)

1. Источник данных: коллекция **`orders`**.
2. Фильтр: при необходимости только оплаченные (`status` = `paid`).
3. **Столбики / линия по дням:** измерение = **дата** из поля **`paidAt`** (группировка по дню или месяцу), мера = **`sum(amount)`** (валовая) или **`sum(netAmount)`** (после комиссии, если поля заполнены).
4. **По серверам:** измерение = **`serverId`**, мера = **`sum(amount)`** или **`sum(netAmount)`** — работает, если в webhook или в **`products.serverId`** задана привязка.
5. **Комиссии платёжки:** мера **`sum(feeAmount)`** по тем же осям, что и выручка; или смотрите **`netAmount`**.

### Расходы (infra_costs)

- **Столбцы / круговая**: сумма **`amount`** по **`role`** или по **`serverId`** (одна валюта в отчёте или два ряда RUB / USD).
- **К одной валюте**: заведите в строках **`usdToRubRate`** или вводите расходы сразу в ₽.
- **Период `daily`**: для сравнения с месячной выручкой приведите к ₽/мес (см. поле **`period`** в таблице выше).

### Доход vs расход по серверу (упрощённо)

- **Доход:** `orders` → `sum(netAmount)` или `sum(amount)` с группировкой по **`serverId`**.
- **Расход:** `infra_costs` → `sum(amount)` с группировкой по **`serverId`** (и тем же периодом, что смотрите в отчёте).
- В одном **комбинированном** графике NocoBase иногда удобнее вынести **два блока** рядом или использовать **SQL / Data source**, если плагин позволяет объединить таблицы по `serverId`.

### Прочее

- **Новые клиенты**: число записей **`customers`** по **`lastSeenAt`** или дате создания.
- **Заказы**: количество **`orders`** по дням, разрез по **`productCode`**.
- **Прокси**: количество **`proxy_instances`** по **`country`** / **`serverId`**.

Точные клики в UI зависят от версии NocoBase и плагина визуализации: **Data visualization** / **Charts** — в блоке укажите коллекцию, поля измерения и меры, затем **выполните запрос** и проверьте превью.

## 6. Политика секретов

- **Не хранить** в NocoBase: пароли прокси, приватные SSH-ключи, `subId`/полные subscription URL с секретами, JWT-секреты бота.
- В **proxy_instances** допускается только **username** + метаданные; пароль остаётся в `data/proxy-links.json` на сервере бота (или будущем секрет-хранилище).
- Роли NocoBase: саппорт без экспорта чувствительных полей; API-ключ бота — только create/list на нужных таблицах.

## 7. Проверка

1. Создать коллекции и поля (в т.ч. новые **`feeAmount`**, **`netAmount`**, **`serverId`** в `orders` и опционально **`serverId`** в `products` / `infra_costs`), выдать API key, прописать `NOCOBASE_*` в окружении бота.
2. Перезапустить бот, открыть мини-апп: при заполненных `products` должны смениться кнопки продления.
3. Смоделировать webhook оплаты с заголовком `x-webhook-secret` — в NocoBase появляются `orders` и обновляется `customers`.
4. Создать прокси — появляется строка в `proxy_instances` без пароля.
